spark在standalone模式下，Master如何实现资源的调度和分配

Created2025-03-28|Updated2025-04-14|技术大数据计算批处理

背景

  • spark集群在 standalone 模式下，Master 通过函数schedule()来刷新资源情况，同时启动 Executor 的调度逻辑，为 Application 分配 Executor 资源

流程梳理

1，调用scheduler()函数

（1）代码

（2）代码解析

  • 代码路径为：org.apache.spark.deploy.master.Master#schedule
  • waitingDrivers 队列只在 cluster 模式下使用，在 client 模式下该队列长度一直为0，所以在 client 模式下，不会对 waitingDrivers 队列进行轮询
  • 使用 Random.shuffle(…) 方法，对活跃的 Worker 列表进行随机打乱，以实现负载均衡
  • 调用startExecutorsOnWorkers()函数，启动 Executor 的调度逻辑，为 Application 分配 Executor 资源

2，调用startExecutorsOnWorkers()函数

（1）代码

（2）代码解析

  • 代码路径为：org.apache.spark.deploy.master.Master#startExecutorsOnWorkers
  • 变量waitingApps为存储了所有等待分配资源的应用程序，Spark Master 按照 FIFO （先来先服务）的顺序尝试为每个应用分配资源。
  • 变量coresPerExecutor为提交spark任务的时候–executor-cores指定的参数，如果不进行指定，默认值为1
  • app.coresLeft数值来源：def coresLeft: Int = requestedCores - coresGranted
    • requestedCores：表示应用程序请求的核心数。也就是说，当用户提交一个 Spark 作业时，用户可以通过配置 spark.executor.cores 来指定每个Executor 使用的核心数，并且还可以通过 spark.executor.instances 来指定 Executor 的数量
    • coresGranted：表示已经为应用程序分配的核心数。在作业运行过程中，Spark 会根据集群资源的可用性和调度策略，决定分配多少核心给每个应用程序。它记录了已经分配给该应用程序的核心数。
    • coresLeft 是一个计算属性（getter 方法），它表示当前应用程序剩余未分配的核心数。其值由 requestedCores - coresGranted 计算得出
  • 变量usableWorkers存储状态为 ALIVE 的 Worker
    • 首先过滤 state 为 ALIVE 的 worker
    • canLaunchExecutor函数会对该 Worker 的剩余内存（剩余内存释放大于等于任务需求的内存memoryReq），剩余 CPU 核心数（是否大于等于任务需求的 CPU 核心数coresReq），以及自定义资源检查（是否满足任务的需求resourceRequirements）
    • 按照可用的核心数 coresFree 降序排列，优先考虑资源多的 Worker。分配资源时会优先考虑资源充足的 Worker。
  • 变量appMayHang是检查是否存在资源不足的情况，导致应用程序可能挂起
    • 只有一个等待应用（即当前只有 app）
    • 应用尚未启动任何 Executor
    • 没有符合条件的 Worker 节点
  • 变量assignedCores为一个数组，计算每个 Worker 应分配给当前应用程序的核心数
    • app 为当前的应用程序
    • usableWorkers 为可用 Worker 列表
    • spreadOutApps 表示是否需要将 Spark 会将应用程序的 Executor 尽量均匀分布到多个 Worker 上
  • 在目标 Worker 上面启动当前 app 对应的 Executor，更新 Worker 和 Application 的状态，记录已分配的核心和 Executor 信息

3，详细分析函数startExecutorsOnWorkers()里面调用的scheduleExecutorsOnWorkers函数

（1）代码

（2）代码解析

  • coresPerExecutor: 每个 Executor 请求的核心数
  • minCoresPerExecutor: 如果 coresPerExecutor 是 None（即未指定），那么默认为 1
  • oneExecutorPerWorker: 表示一个 Worker 里面有几个 Executor ，如果 coresPerExecutor 没有指定，默认为1，coresPerExecutor.isEmpty 为 true，则证明该 Worker 上面只存在一个 Executor ，这个唯一的 Executor 使用了这个 Worker 里面所有的核
  • memoryPerExecutor：每个 Executor 请求的内存（MB）
  • resourceReqsPerExecutor：每个 Executor 请求的额外资源（如 GPU）
  • numUsable：表示可用的 Worker 的数量
  • assignedCores：表示每个 Worker 上为该 app 分配的核心数，例如val assignedCores = [2, 4, 3, 2]
  • assignedExecutors：表示每个 Worker 上已经启动的 Executor 数量，例如初始值为[0, 0, 0, 0, 0]
  • coresToAssign：表示总共可分配的核心数，取 app.coresLeft 和所有 Worker 空闲核心数之和的最小值
    • app.coresLeft: 这是当前应用程序 app 仍然需要的核心数。每个应用程序在启动时会请求一定数量的核心，当资源分配后，这个值会减少
    • usableWorkers.map(_.coresFree).sum：这是所有 可用 Worker 上的空闲核心数的总和。usableWorkers 是一个数组，包含了当前所有可用的 Worker（状态为 ALIVE 且资源满足的 Worker）
    • 因为应用程序最多只能获得它所请求的核心数，而集群中也只能提供有限的空闲资源。因此，我们取这两个数中的较小值，确保：应用程序不会请求超过剩余需求的核心数，集群资源也不会分配超过实际空闲核心数的资源
    • 示例：应用程序 app 还需要 40 个核心（即 app.coresLeft = 40），集群中的可用 Worker 总共有 30 个空闲核心（即 usableWorkers.map(_.coresFree).sum = 30）。var coresToAssign = math.min(40, 30)，此时 coresToAssign 将会被设置为 30，表示应用程序最多只能分配到 30 个核心，因为集群没有足够的空闲资源来满足它的完整需求
  • 解析canLaunchExecutorForApp函数，该函数是判断指定的 Worker 能否为该 app 启动 Executor
    • keepScheduling: 如果剩余的可分配核心数大于或等于每个 Executor 所需的最小核心数，允许继续调度
    • enoughCores: 如果当前 Worker 上空闲的核心数大于或等于每个 Executor 所需的核心数，表示该 Worker 可以启动一个 Executor
    • assignedExecutorNum: 当前 Worker 上已经分配和启动的 Executor 数量
      • 初始生成的assignedExecutors变量为长度为numUsable的空Array，每个初始元素都是0
    • launchingNewExecutor：如果允许每个 Worker 启动多个 Executors，或者当前 Worker 上没有 Executor，那么就启动一个新的 Executor
      • oneExecutorPerWorker = coresPerExecutor.isEmpty，当 coresPerExecutor 没有具体指定，oneExecutorPerWorker为true，仅当 Worker 上尚未启动任何 Executor （assignedExecutorNum == 0）时，才能启动新 Executor。一旦成功启动一个 Executor，即assignedExecutorNum != 0，后续不再允许在该 Worker 上面启动其它 Executor。这样也就意味着该 Worker 上面只启动一个 Executor，该 Executor 会使用该 Worker 为此 app 分配的所有核心数
      • 当 coresPerExecutor 指定了具体的核心数，oneExecutorPerWorker 为 false，表示允许在 Worker 上启动多个 Executor，直到资源耗尽
      • launchingNewExecutor 为false的时候，coresPerExecutor.isEmpty 为 true，该 Worker 上的只允许运行一个 Executor，该 Worker 上已经有一个 Executor 运行，此时无法运行新的 Executor，调度器会尝试 为现有 Executor 分配更多核心（而非启动新 Executor）。因此，判断条件只需关注 核心资源是否足够，而无需重复检查内存、其他资源或 Executor 总数限制
    • assignedMemory：表示在该 Worker 上面已经启动和存在的 Executor 里面使用的总内存
    • enoughMemory：判断该 Worker 上面未使用的总的内存能否在启动一个 Executor
    • assignedResources：将每个 Executor 所需的资源转化位 Map 形式，示例：如果一个 executor 需要 2 个 CPU 核心和 4 GB 内存，并且要为某个 worker 分配 3 个 executor，那么：assignedResources 可能会是：{CPU -> 6, Memory -> 12}
    • resourcesFree：计算和更新当前 Worker 上每种资源的剩余量，示例：如果某个 worker 上有 10 个可用 CPU 核心（CPU -> 10），并且为其分配了 6 个 CPU 核心（assignedResources.getOrElse(CPU, 0) = 6），那么：经过这段代码处理后，resourcesFree 将变成：{CPU -> 4, Memory -> 10}，即该 worker 上剩余的 CPU 核心数为 4
    • enoughResources：将上面计算的 resourcesFree 和 resourceReqsPerExecutor 进行比较，如果每种资源都符合条件，就返回 true，否则返回 false
    • underLimit：如果应用程序的 Executor 数量还没有达到 executorLimit，则继续分配
      • assignedExecutors.sum 表示 本次调度过程中 在所有 Worker 上计划新增的 Executor 数量
      • app.executors.size：当前应用程序已存在的 Executor 总数（调度前已分配的 Executor 数量）
      • app.executorLimit：应用程序的 最大 Executor 数量限制，通常由 spark.executor.instances 或动态资源分配策略（如 spark.dynamicAllocation.maxExecutors）设定
  • freeWorkers：通过 canLaunchExecutorForApp 函数获取所有可以启动 Executor 的 Worker 索引
  • 当 freeWorkers 不为空的时候，对 freeWorkers 的索引进行遍历
    • 当符合 keepScheduling && canLaunchExecutorForApp(pos) 条件的时候，就分配一次 Executor 从总的可分配核数 coresToAssign 里面减去指定的最新分配的核心数 minCoresPerExecutor，同时使用assignedCores(pos) += minCoresPerExecutor更新该 Worker 在 assignedCores 对应索引上已经分配好的 Executor 数据量
    • 当 oneExecutorPerWorker 为 true，即 coresPerExecutor.isEmpty 为 true，Worker 上只能启动 1 个 Executor，设置assignedExecutors(pos) = 1，否则设置 assignedExecutors(pos) += 1
  • spreadOutApps：如果 spreadOutApps 为 true，keepScheduling 为false，keepScheduling && canLaunchExecutorForApp(pos)为 false，会自动跳出对当天 Worker 的调度。意味着应用希望将 Executors 分布到尽可能多的 Worker 上，调度完一个 Worker 后就停止继续为该 Worker 分配更多的 Executor
  • 当对 freeWorkers 里面的每一个索引 Worker 进行调度后，调用freeWorkers = freeWorkers.filter(canLaunchExecutorForApp)，更新 freeWorkers 里面的 Worker 索引，直到 freeWorkers 列表为空
  • 该函数最终返回的是对各个 Worker 已经分配好核心数的列表 assignedCores

4，详细分析函数startExecutorsOnWorkers()里面调用的allocateWorkerResourceToExecutors函数

（1）代码

（2）代码解析

  • 该函数是针对遍历 assignedCores 计算出来的每个 Worker 能分配出来的核心数列表来启动 Executor
  • numExecutors：表示该 Worker 里面可以启动 Executor 的个数
    • assignedCores 为该 Worker 指定的总核心数
    • coresPerExecutor 为用户提交任务指定的一个 Executor 需要使用的 core
    • 如果 coresPerExecutor 没有指定，numExecutors 默认为1，即在该 Worker 上为当前的 app 只启动一个 Executor，该 Executor 会占用所有分配给它的核心（assignedCores）
  • coresToAssign：如果没有指定，则 coresToAssign 直接等于 assignedCores，即将所有分配的核心数赋给一个 Executor
  • 遍历在目前的 Worker 上为该 app 分配的 Executor 个数：numExecutors
    • allocated：通过调用 worker 的 acquireResources 方法来分配资源。app.desc.resourceReqsPerExecutor 是应用程序对于每个 Executor 的资源需求（例如内存、GPU 等）。该方法返回分配给 Executor 的资源。为即将启动的 Executor 在 Worker 节点上分配所需的资源，其中 allocated 变量存储的是成功分配的资源信息
    • exec：为 worker 创建一个 Executor。worker 是目标节点，coresToAssign 是分配给该 Executor 的核心数，allocated 是为该 Executor 分配的其他资源（如内存、GPU 等）。app.addExecutor 方法会将这个 Executor 添加到应用程序中，并返回 Executor 对象
    • 调用 launchExecutor 方法启动 Executor。此方法负责将 Executor 调度到指定的 worker 节点，并执行该 Executor
    • 更新该 app 的状态为 RUNNING

5，详细分析函数allocateWorkerResourceToExecutors里面调用的launchExecutor函数

（1）代码

（2）代码解析

  • addExecutor 是 WorkerInfo 类的一个方法，它将 Executor 注册到 Worker 中。这样，Worker 就知道该 Executor 的存在，并可以为它分配资源
  • 该消息会发送到 Worker 的 endpoint，这是 Worker 接收消息的端点。通过这种方式，Master 告诉 Worker 启动一个新的 Executor 并为其分配资源
  • 发送到 Worker 的 endpoint的消息后，在spark/deploy/worker/Worker.scala里面receive接收LaunchExecutor，进行数据处理
  • 向 Driver 发送消息，通知 Driver 该 Executor 已经添加到指定的 Worker 上，消息接收方为org.apache.spark.deploy.client.StandaloneAppClient.ClientEndpoint.receive
